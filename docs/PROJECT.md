# Spolia Wall — Project Documentation

## Overview

The Spolia Wall is a 3D-visualised architectural installation: three 20m x 1m balcony railings stacked at 3 story heights, filled with hundreds of sourced physical objects from European marketplaces. An AI procurement agent (Antoine) finds, negotiates, buys, and tracks objects. A web app renders the wall in real time — each purchase appears as a coloured block (or a 3D mesh when generated).

**Everything runs on GitHub**: JSON files as database, GitHub Actions for CI/CD, GitHub Pages for hosting.

### Status Pipeline

```
SCOUTED → ASSESSED → PENDING_REVIEW → APPROVED → NEGOTIATING → PURCHASED → SHIPPED → RECEIVED → ALLOCATED → INSTALLED
                                                                                                                  (terminal)
Any non-terminal state → REJECTED (terminal)
```

### Material Categories

| Cat | Material | Fire | Target % of wall |
|-----|----------|------|-------------------|
| CAT-A | Stone, ceramic, brick | RF1 | 40–50% |
| CAT-B | Metal, steel, iron | RF1 | 20–30% |
| CAT-C | Experimental, glass, composite | Needs MDS | 5–15% |
| CAT-D | Digital debris, plastic | RF3 | 3–8% |
| CAT-E | Domestic, wood, textile | RF2 | 10–15% |

### Demo Elements

Elements with `"demo": true` are fictional test data. They render at 60% opacity in the 3D wall and show a "DEMO — FICTIONAL ELEMENT" banner in the detail panel. The HUD counts them separately. They should be removed before production use.

## The 3D Wall

### Packing Algorithm

Slots are generated by `app/generate_slots.mjs` using a **bitmap-based free-form packing algorithm** — no rows, no shelves, no skyline. The result is an irregular organic mosaic like real spolia walls.

**Grid**: 5mm resolution → 4,000 x 200 pixel bitmap per level.

**Phase 1 — Random Placement** (8,000 attempts per level):
1. Generate random dimensions: width 25–350mm, height 25–300mm
2. Depth: 80% normal (20–300mm), 15% medium protrusion (200–500mm), 5% dramatic (400–800mm)
3. Pick random grid position, check bitmap collision, place if clear

**Phase 2 — Gap Fill**: Scan remaining empty cells, fill with infill pieces (max 200x200mm). Pieces <10x10mm = mortar (not tracked).

**Result**: ~2,500–2,800 slots per level, ~8,000 total, 199 unique Y positions — zero visible row structure.

### Three Levels

| Level | Base Height | Y Range | Fire Zone |
|-------|-------------|---------|-----------|
| Level 1 (Ground) | 0mm | 0–1,000mm | Zone 1 |
| Level 2 (First floor) | 3,500mm | 3,500–4,500mm | Zone 2 |
| Level 3 (Second floor) | 7,000mm | 7,000–8,000mm | Zone 3 |

### Rendering

- Scale: 1mm = 0.001 Three.js units (wall is 20 units long)
- Empty slots: wireframe, light grey, 30% opacity
- Filled slots: solid `MeshStandardMaterial` with status/category colour
- Demo elements: 60% opacity (translucent)
- PENDING_REVIEW: yellow emissive pulse
- GLB meshes: loaded async via `GLTFLoader`, scaled to 95% fill, coloured by status
- Fallback: if GLB fails to load, coloured box remains

### Reactivity

The wall auto-updates every 30 seconds:

1. `data.js` fetches elements.json and slots.json from GitHub raw content (cache-busted)
2. `processUpdate()` diffs new data against current state
3. New elements: scale-in animation
4. Status changes: smooth colour transition
5. PENDING_REVIEW: registers yellow pulse animation

Manual refresh: press `R` key.

## Agent Pipeline

Antoine operates through 4 Python scripts in `agent/`. All use stdlib only.

### End-to-End Flow

```
1. SCOUT        Antoine finds a listing online
                → add_element.py creates EL-XXXX record

2. ASSESS       Antoine evaluates dimensions, material, condition
                → update_status.py EL-XXXX ASSESSED

3. REVIEW       If needed, flag for architect approval
                → update_status.py EL-XXXX PENDING_REVIEW --flag-review

4. APPROVE      Architect (or auto) approves
                → update_status.py EL-XXXX APPROVED
                → Mesh generation becomes eligible

5. NEGOTIATE    Antoine contacts seller, negotiates price
                → update_status.py EL-XXXX NEGOTIATING --append-log "..."

6. PURCHASE     Payment confirmed
                → update_status.py EL-XXXX PURCHASED --agreed-price X --shipping Y

7. SHIP         Seller dispatches
                → update_status.py EL-XXXX SHIPPED

8. RECEIVE      Object arrives in Switzerland
                → update_status.py EL-XXXX RECEIVED

9. ALLOCATE     Assign to specific wall slot
                → assign_slot.py EL-XXXX SLOT-L1-042

10. INSTALL     Physically mounted
                → update_status.py EL-XXXX INSTALLED
```

Each step = one git commit. The git log becomes the procurement diary.

## 3D Mesh Generation

### Pipeline

1. Element reaches APPROVED status (or ASSESSED with no review flag)
2. `generate_mesh.py` sends element's best photo to fal.ai TripoSR API
3. API returns a GLB (GL Binary) 3D model (~$0.07 per generation)
4. GLB stored at `data/meshes/EL-XXXX.glb`
5. Element's `mesh_url` field updated
6. Three.js app loads GLB via `GLTFLoader`, scales to element dimensions

### Format

**GLB** (binary glTF) — not OBJ. This is what fal.ai TripoSR outputs and what Three.js GLTFLoader expects. GLB is compact, single-file, includes geometry + materials.

### API Details

```
POST https://fal.run/fal-ai/triposr
Authorization: Key {FAL_API_KEY}

{
  "image_url": "https://...",
  "output_format": "glb",
  "do_remove_background": true,
  "foreground_ratio": 0.9,
  "mc_resolution": 256
}
```

## Composition Rules

Checked automatically when assigning elements to slots:

1. **No same subcategory adjacent** — Two `ceramic_tile` objects cannot be neighbours
2. **No same category adjacent** — Two CAT-A objects cannot be neighbours
3. **CAT-D surrounded** — Digital debris must have CAT-A or CAT-B on all adjacent sides
4. **Depth variation** — Adjacent objects must differ by at least 20mm in depth
5. **Fire zone compliance**:
   - Zone 1 (0–3m): all categories
   - Zone 2 (3–6m): no CAT-D unless encapsulated
   - Zone 3 (6m+): only CAT-A, CAT-B; CAT-C needs material data sheet

Toggle composition overlay on the 3D wall with `C` key. Violations show as red wireframes.

## Deployment

### GitHub Actions (`.github/workflows/deploy.yml`)

Triggers on push to `main` affecting `app/`, `data/`, or the workflow file.

1. Checkout code
2. Setup Node 20
3. `npm ci` in `app/`
4. Copy `data/` → `app/public/data/`
5. `npm run build` (Vite → `app/dist/`)
6. Deploy `app/dist/` to `gh-pages` branch via `peaceiris/actions-gh-pages@v4`

### Local Development

```bash
cd app
npm install
npm run dev    # localhost:5173
```

Data files are in `app/public/data/` for dev mode, fetched from GitHub raw for production.

## Future Considerations

### Scheduling Antoine

Antoine needs to run on a schedule to continuously source objects. Options:
- **GitHub Actions cron**: Run sourcing scripts on a schedule (e.g., every 6 hours)
- **Claude Code scheduled agents**: Use Claude's trigger/schedule system
- **External scheduler**: Cloud function or cron job that invokes agent scripts

### Supplier Contact Management

Antoine needs to:
- Track seller communication history (currently in `negotiation_log` field)
- Follow up on unanswered messages
- Handle multi-object negotiations with the same seller
- Manage shipping coordination across multiple purchases

Potential approach: extend elements.json with a `sellers.json` database that tracks seller profiles, communication history, reliability scores, and bulk deal status.

### Workflow Visualization

Two possible approaches:
1. **In the 3D wall**: Add a mode that shows Antoine's current actions as animated particles or overlays on the wall itself — scouting targets glow, negotiations pulse, etc.
2. **Separate dashboard**: A second HTML page (e.g., `app/dashboard.html`) that shows Antoine's pipeline as a Kanban board, timeline, or flowchart. Could live alongside the 3D wall in the same deployment.

### Payment Tracking

- CHF conversion rates at point of purchase
- Running budget vs. target
- Cost per m² of wall coverage
- Category budget allocation tracking

### Quality Assurance

- Photo verification workflow (seller photo vs. received object)
- Dimension verification (estimated vs. actual measurements)
- Condition grading system
- Return/rejection tracking with seller feedback
